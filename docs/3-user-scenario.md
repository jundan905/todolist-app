## 1. 개요

### 1.1 문서 목적

이 문서는 todolist-app의 핵심 유스케이스(UC-01 ~ UC-08)를 실제 사용자 관점에서 구체적인 시나리오로 서술한다. 개발팀, 디자인팀, QA팀이 공통된 맥락으로 사용자 경험을 이해하고, 기능 구현 및 테스트 설계의 기준으로 활용하는 것을 목적으로 한다.

각 시나리오는 특정 페르소나의 실생활 맥락에서 출발하며, 사용자와 시스템 간의 단계별 상호작용을 "사용자가 ~한다 → 시스템이 ~한다" 형식으로 기술한다. 정상 흐름(Happy Path)과 예외 흐름(Exception Path)을 모두 다루어 엣지 케이스까지 포괄한다.

### SCN-01: 신규 가입 후 첫 할일 등록 (UC-01, UC-02, UC-04)

#### 페르소나

**김민준 (28세, 스타트업 마케터)**

#### 상황

오늘 팀장에게 새 SNS 캠페인 기획서를 다음 주 월요일까지 제출하라는 지시를 받았다. 지금까지 메모장과 종이 메모로 업무를 관리해 왔는데, 이번 기회에 체계적인 할일 관리 앱을 써보기로 한다. 동료가 추천해 준 todolist-app에 접속한다.

#### 목표

서비스에 처음 가입하고, 로그인한 뒤, 캠페인 기획서 작성 할일을 등록한다.

---

#### 정상 흐름 (Happy Path)

**[단계 1] 회원가입 페이지 접근**
1. 김민준이 todolist-app 웹사이트에 접속한다.
2. 시스템이 로그인 화면을 표시하고, 하단에 "계정이 없으신가요? 회원가입" 링크를 제공한다.
3. 김민준이 "회원가입" 링크를 클릭한다.
4. 시스템이 회원가입 폼(이메일, 비밀번호, 이름 입력란)을 표시한다.

**[단계 2] 회원가입 정보 입력 및 제출**
5. 김민준이 이메일 입력란에 `minjun.kim@startup.com`을 입력한다.
6. 김민준이 비밀번호 입력란에 `Startup!23`을 입력한다. (대문자, 소문자, 숫자, 특수문자 포함, 9자)
7. 김민준이 이름 입력란에 `김민준`을 입력한다.
8. 김민준이 "가입하기" 버튼을 클릭한다.
9. 시스템이 이메일 형식, 비밀번호 복잡도, 이름 유효성을 검사하고 모두 통과한다.
10. 시스템이 계정을 생성하고 "회원가입이 완료되었습니다" 메시지와 함께 로그인 페이지로 이동시킨다.

**[단계 3] 로그인**
11. 김민준이 이메일 입력란에 `minjun.kim@startup.com`을 입력한다.
12. 김민준이 비밀번호 입력란에 `Startup!23`을 입력한다.
13. 김민준이 "로그인" 버튼을 클릭한다.
14. 시스템이 이메일과 비밀번호를 검증하고, 유효한 인증 토큰(Access Token)을 발급한다.
15. 시스템이 할일 목록 메인 화면으로 이동시킨다. 등록된 할일이 없으므로 빈 목록과 "첫 번째 할일을 추가해보세요" 안내 문구를 표시한다.

**[단계 4] 할일 생성**
16. 김민준이 "할일 추가" 버튼을 클릭한다.
17. 시스템이 할일 생성 폼(제목, 설명, 시작일, 종료일 입력란)을 표시한다.
18. 김민준이 제목 입력란에 `SNS 캠페인 기획서 작성`을 입력한다.
19. 김민준이 설명 입력란에 `2분기 인스타그램/유튜브 캠페인 전략 포함`을 입력한다.
20. 김민준이 시작일을 오늘 날짜(`2026-03-31`)로 선택한다.
21. 김민준이 종료일을 다음 주 월요일(`2026-04-06`)로 선택한다.
22. 김민준이 "저장" 버튼을 클릭한다.
23. 시스템이 입력값을 검증한다. 종료일(4월 6일)이 시작일(3월 31일)보다 이후이므로 조건을 충족한다.
24. 시스템이 할일을 생성하고, 오늘이 시작일 범위 내이므로 상태를 `IN_PROGRESS`로 계산한다.
25. 시스템이 할일 목록 화면으로 돌아가며, 방금 생성한 `SNS 캠페인 기획서 작성` 항목이 `진행 중` 상태로 목록에 표시된다.

#### 기대 결과

- 김민준의 계정이 정상적으로 생성된다.
- 로그인 후 인증 토큰이 브라우저에 저장되어 이후 요청에 자동으로 사용된다.
- `SNS 캠페인 기획서 작성` 할일이 `IN_PROGRESS` 상태로 목록에 나타난다.
- 종료일인 4월 6일 이전에 완료 처리하면 `COMPLETED`로, 이후에 처리하면 `LATE_COMPLETED`로 상태가 전환된다.

#### 예외 흐름 (Exception Path)

**[예외 1] 이미 가입된 이메일로 회원가입 시도**

- 상황: 김민준이 이전에 다른 기기로 이미 가입한 이메일 `minjun.kim@startup.com`으로 재가입을 시도한다.
- 단계 8에서 "가입하기" 버튼을 클릭한다.
- 시스템이 해당 이메일이 이미 등록되어 있음을 감지하고 가입을 거부한다.
- 시스템이 폼 상단에 "이미 사용 중인 이메일 주소입니다. 로그인 페이지로 이동하거나 다른 이메일을 입력해주세요" 오류 메시지를 표시한다.
- 이메일 입력란이 빨간 테두리로 강조되고, 비밀번호 입력값은 초기화된다.

**[예외 2] 비밀번호 복잡도 규칙 미충족**

- 상황: 김민준이 기억하기 쉬운 비밀번호 `startup123`을 입력한다. (대문자와 특수문자 없음)
- 단계 8에서 "가입하기" 버튼을 클릭한다.
- 시스템이 비밀번호 복잡도 검사를 수행하고 규칙 미충족을 감지한다.
- 시스템이 "비밀번호는 8~20자이며, 영문 대문자, 소문자, 숫자, 특수문자(!@#$%^&*)를 각각 1자 이상 포함해야 합니다" 오류 메시지를 표시한다.
- 비밀번호 입력란이 초기화되고 다시 입력을 요청한다.

**[예외 3] 비밀번호가 20자를 초과하는 경우**

- 상황: 김민준이 보안을 강조하여 `StartupMarketer!2026Plus` (24자)를 입력한다.
- 시스템이 비밀번호 길이 초과를 감지하고 "비밀번호는 최대 20자까지 입력 가능합니다" 오류 메시지를 표시한다.

**[예외 4] 이메일 형식이 올바르지 않은 경우**

- 상황: 김민준이 이메일 입력란에 `minjun.kim`(@ 누락)을 입력한다.
- 시스템이 이메일 형식 검증에서 실패하고 "올바른 이메일 형식을 입력해주세요 (예: example@domain.com)" 메시지를 표시한다.

**[예외 5] 할일 생성 시 종료일이 시작일보다 이전인 경우**

- 상황: 김민준이 실수로 시작일을 `2026-04-06`, 종료일을 `2026-03-31`로 입력한다.
- 단계 22에서 "저장" 버튼을 클릭한다.
- 시스템이 날짜 유효성 검사에서 종료일이 시작일보다 이전임을 감지한다.
- 시스템이 "종료일은 시작일과 같거나 이후 날짜여야 합니다" 오류 메시지를 표시한다.
- 입력 폼은 유지되고 날짜 입력란이 강조 표시된다.

### SCN-02: 오늘 해야 할 일 확인 및 완료 처리 (UC-02, UC-05, UC-08)

#### 페르소나

**김민준 (28세, 스타트업 마케터)**

#### 상황

월요일 아침, 김민준이 출근 직후 오늘 처리해야 할 업무를 확인하려 한다. 지난주에 여러 할일을 등록해 둔 상태다. 업무를 끝낼 때마다 앱에서 완료 표시를 해두는 습관이 있다. 오전 중에 경쟁사 분석 자료 조사를 마치고 완료 처리하려 한다.

#### 목표

로그인 후 현재 진행 중인 할일 목록을 확인하고, 완료한 업무를 완료 처리한다.

---

#### 정상 흐름 (Happy Path)

**[단계 1] 로그인**
1. 김민준이 todolist-app에 접속한다.
2. 시스템이 로그인 화면을 표시한다.
3. 김민준이 이메일 `minjun.kim@startup.com`과 비밀번호 `Startup!23`을 입력하고 "로그인" 버튼을 클릭한다.
4. 시스템이 인증을 처리하고 새로운 인증 토큰을 발급한다.
5. 시스템이 할일 목록 메인 화면으로 이동시킨다.

**[단계 2] 진행 중인 할일 목록 확인**
6. 김민준이 상태 필터에서 "진행 중(IN_PROGRESS)"을 선택한다.
7. 시스템이 현재 날짜 기준으로 `startDate <= 오늘 <= dueDate`이고 미완료인 할일만 필터링하여 목록을 반환한다.
8. 목록에는 `경쟁사 분석 자료 조사`, `SNS 캠페인 기획서 작성`, `주간 보고서 준비` 등 여러 항목이 표시된다.
9. 김민준이 각 할일의 제목과 종료일을 확인하며 오늘 처리할 항목을 파악한다.

**[단계 3] 할일 완료 처리**
10. 김민준이 `경쟁사 분석 자료 조사` 항목 옆의 완료 버튼(체크박스 또는 완료 처리 버튼)을 클릭한다.
11. 시스템이 해당 할일의 `isCompleted`를 `true`로 변경하고, 서버의 현재 시각을 `completedAt`으로 자동 기록한다.
12. 시스템이 오늘이 해당 할일의 종료일 이내임을 확인하여 상태를 `COMPLETED`로 계산한다.
13. 해당 항목이 목록에서 시각적으로 완료 상태(취소선 또는 색상 변경)로 표시되거나, `IN_PROGRESS` 필터 화면에서 사라진다.
14. 김민준이 완료 처리가 반영된 것을 확인한다.

#### 기대 결과

- 로그인 후 `IN_PROGRESS` 필터로 오늘 처리해야 할 업무만 빠르게 확인할 수 있다.
- 완료 처리 즉시 화면에 반영되어 남은 업무를 직관적으로 파악할 수 있다.
- `completedAt`이 서버 시각으로 자동 기록되어 별도로 완료 시간을 입력할 필요가 없다.

---

#### 예외 흐름 (Exception Path)

**[예외 1] 인증 토큰 만료 상태에서 완료 처리 시도**

- 상황: 김민준이 아침에 로그인한 후 회의 등으로 자리를 비웠다가 1시간 이상 지난 후 돌아와 완료 처리를 시도한다.
- 단계 10에서 완료 버튼을 클릭한다.
- 시스템이 인증 토큰이 만료되었음(발급 후 1시간 초과)을 감지한다.

### SCN-03: 마감일 지난 할일 확인 및 뒤늦게 완료 처리 (UC-05, UC-06, UC-08)

#### 페르소나

**이수진 (19세, 공대 1학년 대학생)**

#### 상황

이수진은 지난주에 물리학 과제를 `2026-03-28`(지난 금요일)까지 제출해야 했는데, 중간고사 준비에 치여서 잊어버렸다. 오늘(`2026-03-31`)이 되어서야 할일 목록을 확인하다 해당 과제를 발견했다. 교수님이 늦게라도 제출을 받아주기로 해서 오늘 과제를 완료하고 앱에서도 완료 처리를 하려 한다.

#### 목표

기한이 지난 `FAILED` 상태의 할일을 확인하고, 뒤늦게 완료 처리하여 `LATE_COMPLETED` 상태로 전환한다.

---

#### 정상 흐름 (Happy Path)

**[단계 1] 기한 초과 할일 목록 확인**
1. 이수진이 앱에 로그인된 상태로 할일 목록 화면을 연다.
2. 이수진이 상태 필터에서 "미완료(FAILED)"를 선택한다.
3. 시스템이 현재 날짜(`2026-03-31`)를 기준으로 종료일이 지났으나 완료되지 않은 할일만 필터링하여 반환한다.
4. 목록에 `물리학 과제 제출 (마감: 2026-03-28)` 항목이 `FAILED` 상태로 표시된다.

**[단계 2] 할일 상세 내용 확인**
5. 이수진이 `물리학 과제 제출` 항목을 클릭하여 상세 화면으로 이동한다.
6. 시스템이 해당 할일의 상세 정보(제목, 설명, 시작일 `2026-03-24`, 종료일 `2026-03-28`, 현재 상태 `FAILED`)를 표시한다.
7. 이수진이 과제 내용과 마감일을 다시 한번 확인한다.

**[단계 3] 뒤늦게 완료 처리 (FAILED → LATE_COMPLETED)**
8. 이수진이 상세 화면에서 완료 처리 버튼을 클릭한다.
9. 시스템이 해당 할일의 `isCompleted`를 `true`로 변경하고, 서버의 현재 시각(`2026-03-31 오후 2시 30분`)을 `completedAt`으로 자동 기록한다.
10. 시스템이 상태 계산 알고리즘을 실행한다. `completedAt(2026-03-31)`이 `dueDate(2026-03-28)`보다 이후이므로 상태를 `LATE_COMPLETED`로 결정한다.
11. 상세 화면에서 할일 상태가 "기한 초과 완료(LATE_COMPLETED)"로 업데이트되어 표시된다.
12. 이수진이 목록 화면으로 돌아오면 해당 항목이 더 이상 `FAILED` 필터에 표시되지 않는다.

#### 기대 결과

- 기한이 지난 할일도 완료 처리가 가능하며, `LATE_COMPLETED` 상태로 이력이 남는다.
- `completedAt`이 종료일보다 늦으면 자동으로 `LATE_COMPLETED`로 분류되어 기한 준수 여부를 명확히 알 수 있다.
- `FAILED` 필터 목록에서 해당 항목이 사라져 아직 처리해야 할 미완료 항목만 남는다.

---

#### 예외 흐름 (Exception Path)

**[예외 1] FAILED 필터 목록이 비어있는 경우**

- 상황: 이수진이 `FAILED` 필터를 선택했는데 기한 초과된 미완료 항목이 없다.
- 시스템이 조건을 만족하는 할일이 없으므로 빈 목록과 함께 "기한이 초과된 미완료 할일이 없습니다" 안내 메시지를 표시한다.

**[예외 2] 완료 취소 시 LATE_COMPLETED → FAILED로 전환**

- 상황: 이수진이 완료 처리 후 과제 내용이 잘못됐음을 발견하고 완료를 취소하려 한다.
- 이수진이 `LATE_COMPLETED` 상태의 항목에서 완료 취소 버튼을 클릭한다.
- 시스템이 `isCompleted`를 `false`로, `completedAt`을 `null`로 초기화한다.
- 시스템이 상태를 재계산한다. 오늘(`2026-03-31`)이 종료일(`2026-03-28`)보다 이후이고 `isCompleted`가 `false`이므로 상태를 `FAILED`로 결정한다.

- 시스템이 "로그인이 필요한 서비스입니다" 메시지와 함께 로그인 화면으로 이동시킨다.

**[예외 4] 다른 사용자의 할일에 완료 처리 시도**

### SCN-04: 할일 내용 수정 (UC-06, UC-07)

#### 페르소나

**이수진 (19세, 공대 1학년 대학생)**

#### 상황

이수진이 지난주에 등록한 `운영체제 중간고사 준비` 할일의 종료일을 `2026-04-10`으로 잡았는데, 교수님이 시험 일정을 `2026-04-15`로 변경했다. 또한 공부 범위에 대한 내용도 구체적으로 추가하고 싶다. 앱에서 해당 할일을 찾아 종료일과 설명을 수정하려 한다.

#### 목표

기존 할일의 종료일을 변경하고 상세 설명을 추가한다.

---

#### 정상 흐름 (Happy Path)

**[단계 1] 수정할 할일 상세 조회**
1. 이수진이 할일 목록 화면에서 `운영체제 중간고사 준비` 항목을 찾는다.
2. 이수진이 해당 항목을 클릭하여 상세 화면으로 이동한다.
3. 시스템이 해당 할일의 상세 정보(제목, 설명, 시작일 `2026-04-01`, 종료일 `2026-04-10`, 현재 상태 `UPCOMING`)를 표시한다.
4. 이수진이 현재 등록된 종료일(`2026-04-10`)과 설명 내용을 확인한다.

**[단계 2] 수정 화면으로 이동**
5. 이수진이 상세 화면에서 "수정" 버튼을 클릭한다.
6. 시스템이 기존 입력값이 채워진 수정 폼을 표시한다. 제목(`운영체제 중간고사 준비`), 시작일(`2026-04-01`), 종료일(`2026-04-10`)이 이미 입력되어 있다.

**[단계 3] 내용 수정 및 저장**
7. 이수진이 종료일을 `2026-04-10`에서 `2026-04-15`로 변경한다.
8. 이수진이 설명 입력란에 `3장 프로세스 스케줄링, 4장 메모리 관리, 5장 파일 시스템 중점 공부`를 입력한다.
9. 이수진이 "저장" 버튼을 클릭한다.
10. 시스템이 수정된 내용의 유효성을 검사한다. 종료일(`2026-04-15`)이 시작일(`2026-04-01`)보다 이후이므로 조건을 충족한다.
11. 시스템이 할일 정보를 업데이트하고 `updatedAt`을 현재 시각으로 자동 갱신한다.
12. 시스템이 수정된 정보로 상세 화면을 다시 표시한다. 변경된 종료일(`2026-04-15`)과 새 설명이 반영되어 표시된다.

#### 기대 결과

- 할일의 종료일과 설명이 성공적으로 수정된다.
- `updatedAt`이 수정 시각으로 자동 갱신되어 최종 변경 시점을 추적할 수 있다.
- 상태(`UPCOMING`)는 변경된 날짜 기준으로 재계산되며, 오늘(`2026-03-31`)이 아직 시작일(`2026-04-01`) 이전이므로 그대로 `UPCOMING`을 유지한다.

---

#### 예외 흐름 (Exception Path)

### SCN-05: 상태별 필터링으로 할일 현황 파악 (UC-05)

#### 페르소나

**박영철 (45세, 카페 운영자)**

#### 상황

박영철은 매주 월요일 아침 카운터 PC에서 이번 주 카페 운영 관련 할일들을 점검하는 루틴이 있다. 오늘도 커피 한 잔을 마시며 앱을 열어 이번 주 처리해야 할 업무, 아직 시작 전인 업무, 그리고 지난주에 처리 못한 업무를 분류해서 확인하고 싶다.

#### 목표

상태별 필터를 활용하여 카페 운영 관련 할일의 전체 현황을 파악한다.

---

#### 정상 흐름 (Happy Path)

**[단계 1] 전체 할일 목록 조회**
1. 박영철이 로그인된 상태에서 할일 목록 화면을 연다.
2. 시스템이 필터 없이 박영철 본인의 전체 할일 목록을 `createdAt` 내림차순(최신 등록 순)으로 표시한다. 기본적으로 한 페이지에 20개를 표시한다.
3. 박영철이 전체 현황을 훑어본다.

**[단계 2] 진행 중인 업무 확인 (IN_PROGRESS 필터)**
4. 박영철이 상태 필터에서 "진행 중"을 선택한다.
5. 시스템이 오늘 날짜(`2026-03-31`) 기준으로 `startDate <= 오늘 <= dueDate`이고 미완료인 할일만 필터링한다.
6. `커피 원두 발주 (마감: 2026-04-02)`, `냉장고 정기 청소 (마감: 2026-04-01)` 등 이번 주 처리해야 할 항목들이 표시된다.
7. 박영철이 마감일이 임박한 항목부터 처리 순서를 확인한다.

**[단계 3] 시작 전 업무 확인 (UPCOMING 필터)**
8. 박영철이 상태 필터를 "시작 전(UPCOMING)"으로 변경한다.
9. 시스템이 오늘 날짜보다 `startDate`가 이후인 미완료 할일만 반환한다.
10. `에어컨 필터 교체 (시작: 2026-04-10, 마감: 2026-04-15)` 등 아직 시작일이 안 된 항목이 표시된다.
11. 박영철이 다음 주 이후 예정된 업무를 미리 파악한다.

**[단계 4] 기한 종료된 전체 항목 확인 (CLOSED 필터)**
12. 박영철이 상태 필터를 "종료됨(CLOSED)"으로 변경한다.
13. 시스템이 종료일(`dueDate`)이 오늘 날짜보다 이전인 모든 항목(`FAILED`, `LATE_COMPLETED`, 기한 초과 `COMPLETED` 포함)을 한 번에 필터링하여 반환한다.
14. `지난주 발주서 제출 (마감: 2026-03-27, 상태: COMPLETED)`, `영수증 정리 (마감: 2026-03-28, 상태: FAILED)` 등 종료된 항목들이 표시된다.
15. 박영철이 `FAILED` 항목을 확인하고 뒤늦게 처리하거나 다음 주 일정에 반영할지 판단한다.

**[단계 5] 정렬 기준 변경**
16. 박영철이 목록 정렬을 "종료일 오름차순"으로 변경한다.
17. 시스템이 `dueDate` 기준 오름차순으로 목록을 재정렬하여 마감이 가까운 항목이 상단에 표시된다.

#### 기대 결과

- 상태별 필터를 통해 지금 당장 해야 할 업무, 앞으로 해야 할 업무, 지난 업무를 명확하게 구분하여 파악할 수 있다.
- `CLOSED` 필터를 통해 기한이 지난 모든 항목을 한 번에 확인할 수 있다.
- 정렬 기능으로 우선순위에 따라 목록을 재배열할 수 있다.

#### 예외 흐름 (Exception Path)

**[예외 1] 잘못된 필터 값 입력 (직접 URL 파라미터 조작)**

- 상황: 박영철의 기술에 익숙한 직원이 URL의 status 파라미터를 `UNKNOWN`으로 직접 변경하여 접근한다.
- 시스템이 허용되지 않는 status 값을 감지한다.
- 시스템이 "유효하지 않은 필터 조건입니다. 올바른 상태 값(UPCOMING, IN_PROGRESS, COMPLETED, LATE_COMPLETED, FAILED, CLOSED)을 사용해주세요" 오류 메시지를 반환한다.

**[예외 2] 잘못된 페이지 번호 요청**

- 상황: URL에서 페이지 번호를 `0` 또는 음수(`-1`)로 설정하여 요청한다.
- 시스템이 유효하지 않은 페이지 번호를 감지한다.
- 시스템이 "페이지 번호는 1 이상의 정수여야 합니다" 오류 메시지를 반환한다.

**[예외 3] 최대 페이지 크기 초과 요청**

- 상황: URL에서 한 페이지 조회 개수(`limit`)를 `150`으로 설정하여 요청한다.
- 시스템이 허용 최대값(100개)을 초과했음을 감지한다.
- 시스템이 "한 번에 조회 가능한 최대 개수는 100개입니다" 오류 메시지를 반환한다.

**[예외 4] 필터 결과가 빈 목록인 경우**

- 상황: 박영철이 `UPCOMING` 필터를 선택했는데 시작 전인 할일이 하나도 없다.
- 시스템이 조건을 만족하는 할일이 없으므로 빈 배열과 함께 "해당 조건에 맞는 할일이 없습니다" 안내 메시지를 표시한다.

**[예외 5] 로그인 없이 목록 조회 시도**

- 상황: 박영철이 즐겨찾기로 저장된 할일 목록 URL에 직접 접근하지만 로그인 세션이 없는 상태다.
- 시스템이 인증 토큰이 없음을 감지한다.
- 시스템이 "로그인이 필요합니다" 메시지와 함께 로그인 화면으로 이동시킨다.

### SCN-06: 로그아웃 및 재로그인 (UC-03, UC-02)

#### 페르소나

**박영철 (45세, 카페 운영자)**

#### 상황

카페 카운터 PC는 박영철 혼자만 사용하지 않는다. 파트타임 직원인 지은이 가끔 PC를 사용한다. 박영철은 자신의 할일 목록을 다른 직원이 보거나 수정하는 것을 원치 않아, 자리를 비울 때마다 로그아웃하는 습관을 들이고 있다. 오늘도 외출 전에 로그아웃을 하고, 돌아온 뒤 다시 로그인하려 한다.

#### 목표

현재 로그인 상태에서 안전하게 로그아웃하고, 나중에 다시 로그인하여 할일 목록을 정상적으로 이용한다.

---

#### 정상 흐름 (Happy Path)

**[단계 1] 로그아웃**
1. 박영철이 할일 목록 화면 우측 상단의 계정 메뉴를 클릭한다.
2. 시스템이 드롭다운 메뉴에 "로그아웃" 옵션을 표시한다.
3. 박영철이 "로그아웃"을 클릭한다.
4. 시스템이 서버에 로그아웃 요청을 전송하고, 현재 인증 토큰이 유효함을 확인한다.
5. 시스템이 브라우저에 저장된 인증 토큰을 삭제한다.
6. 시스템이 "로그아웃되었습니다" 메시지를 표시하며 로그인 화면으로 이동시킨다.
7. 이후 파트타임 직원 지은이 PC를 사용하더라도 박영철의 할일 목록에 접근할 수 없다.

**[단계 2] 재로그인**
8. 박영철이 외출 후 돌아와 로그인 화면에서 이메일 `youngchul.park@cafe.com`과 비밀번호 `Cafe!1234`를 입력한다.
9. 박영철이 "로그인" 버튼을 클릭한다.
10. 시스템이 이메일과 비밀번호를 검증하고, 새로운 인증 토큰을 발급한다.
11. 시스템이 할일 목록 메인 화면으로 이동시킨다.
12. 박영철이 이전에 등록해 둔 할일 목록이 그대로 유지되어 있음을 확인한다.

#### 기대 결과

- 로그아웃 후 인증 토큰이 브라우저에서 완전히 제거되어 다른 사람이 접근할 수 없다.
- 재로그인 시 새로운 인증 토큰이 발급되고 기존 할일 데이터는 그대로 유지된다.
- 로그아웃과 재로그인 과정에서 할일 데이터의 손실이 없다.

---

#### 예외 흐름 (Exception Path)

**[예외 1] 이미 만료된 토큰으로 로그아웃 시도**

- 상황: 박영철이 아침에 로그인한 후 장시간 자리를 비워 인증 토큰이 자동 만료된 상태(`발급 후 1시간 초과`)에서 로그아웃 버튼을 클릭한다.
- 시스템이 인증 토큰이 이미 만료되었음을 감지한다.
- 시스템이 "로그인 세션이 이미 만료되었습니다" 메시지를 표시하며 로그인 화면으로 이동시킨다.
- 브라우저의 만료된 토큰도 함께 삭제한다.

**[예외 2] 만료된 토큰으로 할일 목록 접근 시도**

- 상황: 파트타임 직원 지은이 토큰이 만료된 것을 모르고 브라우저의 즐겨찾기로 할일 목록 URL에 직접 접근한다.
- 시스템이 인증 토큰이 만료되었음을 감지하고 접근을 거부한다.
- 시스템이 "로그인 세션이 만료되었습니다. 다시 로그인해주세요" 안내와 함께 로그인 화면으로 이동시킨다.

**[예외 3] 재로그인 시 비밀번호 오입력**

- 상황: 박영철이 재로그인 시 비밀번호를 `cafe!1234`(대문자 C 없이)로 잘못 입력한다.
- 시스템이 비밀번호 불일치를 감지하고 로그인을 거부한다.
- 시스템이 "이메일 또는 비밀번호가 올바르지 않습니다" 오류 메시지를 표시한다. (보안상 어느 항목이 틀렸는지 명시하지 않는다.)
- 비밀번호 입력란이 초기화되고 다시 입력을 요청한다.

**[예외 4] 재로그인 시 존재하지 않는 이메일 입력**

- 상황: 박영철이 이메일 주소를 `youngchul@cafe.com`(`.park` 누락)으로 잘못 입력한다.
- 시스템이 해당 이메일이 가입된 계정이 없음을 감지하고 로그인을 거부한다.
- 시스템이 "이메일 또는 비밀번호가 올바르지 않습니다" 오류 메시지를 표시한다. (보안상 이메일 존재 여부를 노출하지 않는다.)

**[예외 5] 로그아웃 후 브라우저 뒤로가기로 이전 페이지 접근 시도**

- 상황: 로그아웃 후 브라우저의 뒤로가기 버튼을 눌러 이전에 보던 할일 목록 화면으로 돌아가려 한다.
- 시스템이 해당 페이지 접근 시 인증 토큰이 없음을 감지한다.
- 시스템이 페이지를 로드하지 않고 로그인 화면으로 리다이렉트한다.
- 이미 삭제된 인증 토큰으로는 어떠한 데이터도 조회되지 않는다.
